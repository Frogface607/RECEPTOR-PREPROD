## ВВЕДЕНИЕ

### Что такое IIKO API?

IIKO предоставляет доступ к нескольким программным интерфейсам (API, Application Programming Interface), позволяющим подключать внешние системы и автоматизировать работу с данными.

### Основные типы API IIKO:

1. **iikoFront API** - для разработки плагинов кассовой системы
2. **iikoCloud API** - облачное API для получения информации о ресторанах, заказах, меню
3. **iikoCard API** - для работы с бонусно-депозитной системой
4. **iiko.biz API** - внешнее API (постепенно переходит на Cloud API)
5. **iikoTransport** - транспортный слой для интеграций

### Контакты поддержки:
- **Email**: api@iiko.ru
- **Help Center**: help.iiko.ru
- **GitHub**: front.api.sdk

## ПОЛУЧЕНИЕ ИНФОРМАЦИИ ИЗ IIKO

### iikoCloud API

#### Создание API-ключа

**Шаг 1: Получение API-ключа**
1. Откройте iikoOffice
2. Перейдите в раздел **Обмен данными → Настройка iikoTransport**
3. Произойдет редирект на страницу настройки IikoTransport
4. В разделе **API интеграции** в блоке **Интеграции** нажмите **Добавить**
5. Укажите **Имя api логина** (например, "Receptor", "Analytics")
6. API-ключ будет сгенерирован автоматически - **скопируйте его**
7. Установите флаг **Активный**

**Шаг 2: Выбор организации**
1. В настройках Cloud API откройте вкладку **Организации**
2. Выберите нужную организацию
3. Скопируйте имя **терминальной группы**, на которую будут приходить данные

#### Альтернативный способ (через iikoWeb):
1. Зайдите на iikoweb.ru
2. Перейдите в раздел **Настройки Cloud API**
3. Скопируйте API ключ

### Основные методы получения данных

#### 1. **Получение списка организаций и ресторанов**

```
GET /api/v1/organizations
Headers: Authorization: Bearer {API_KEY}
```

Возвращает:
- ID организаций
- Названия точек
- Адреса
- ID терминальных групп

#### 2. **Получение меню и номенклатуры**

```
GET /api/v1/menu
Parameters:
- organizationId
- includeArchived (опционально)
```

Возвращает:
- Структуру блюд и товаров
- Цены
- Категории
- Модификаторы
- Внешние идентификаторы (артикулы, коды)

#### 3. **Получение заказов**

```
GET /api/v1/orders
Parameters:
- organizationId
- dateFrom
- dateTo
- statuses (опционально)
```

#### 4. **Получение остатков на складе**

```
GET /api/v1/stock
Parameters:
- organizationId
- includeArchived (опционально)
```

Возвращает данные об остатках товаров на складах.

#### 5. **Получение информации о сотрудниках**

```
GET /api/v1/employees
Parameters:
- organizationId
```

Возвращает:
- ID сотрудников
- ФИО
- Должности
- Статусы (активный/неактивный)

## РАБОТА С ТЕХКАРТАМИ

### Что такое техкарта?

Технологическая карта (техкарта) - это документ, в котором задается:
- Норма закладки ингредиентов
- Состав блюда
- Технология приготовления
- Показатели качества
- Требования к оформлению и подаче

Благодаря техкартам при продаже блюда автоматически списываются ингредиенты со склада.

### Структура техкарты в iikoOffice

1. **Вкладка "Технологические карты"** в карточке номенклатуры (блюда)

2. **Основные параметры:**
   - Дата начала действия
   - Дата окончания действия
   - Наименование продукта (ингредиент)
   - Количество
   - Единица измерения
   - Себестоимость

3. **Состав блюда:**
   - Основные ингредиенты
   - Заготовки (полуфабрикаты)
   - Модификаторы

4. **Технология приготовления:**
   - Описание процесса
   - Особенности
   - Требования к качеству

5. **Показатели качества:**
   - Требования к оформлению
   - Особенности подачи
   - Реализации

6. **Фотография блюда** - для печатных форм

### Создание и редактирование техкарты через API

**Методы IIKO Front API для работы с техкартами:**

```
// Получить техкарту для блюда
GetTechnologicalCard(dishId)

// Создать новую техкарту
CreateTechnologicalCard(dishId, ingredients, startDate, endDate)

// Обновить существующую техкарту
UpdateTechnologicalCard(cardId, ingredients)

// Удалить техкарту
DeleteTechnologicalCard(cardId)
```

### Два варианта создания/изменения техкарты

**1. Редактировать существующую техкарту**
- Используется, если нужно перейти на новую версию техкарты
- Все списания переересчитываются
- Применяется ко всем продажам с момента редактирования

**2. Создать новую техкарту**
- Используется, если нужно изменить состав с определенной даты
- Старые списания остаются без изменений
- **Важно!** Указать дату начала действия не позже, чем начали продавать это блюдо

### Обязательные требования для корректной работы техкарт

1. **Все компоненты должны быть в справочнике товаров:**
   - Основные ингредиенты (товары)
   - Заготовки (полуфабрикаты)
   - Модификаторы (с пустой техкартой)

2. **Единицы измерения:**
   - Должны быть правильно настроены
   - Вес указан для каждой единицы
   - Фасовки заполнены (если применимо)

3. **Даты действия:**
   - Дата начала не позже, чем начало продаж
   - Даты не должны пересекаться с предыдущими версиями

4. **Методы списания:**
   - "Списывать ингредиенты" - по техкарте
   - "Списывать готовое блюдо" - целиком блюдо

### Частые ошибки при создании техкарт

1. **Неправильная дата начала действия** - техкарта с датой позже начала продаж приводит к восклицательным знакам (!) в справочнике

2. **Отсутствие ингредиентов в справочнике** - все компоненты должны быть созданы заранее

3. **Несовпадение единиц измерения** - единица в техкарте должна быть определена в карточке товара

4. **Пересечение дат** - создание техкарт с пересекающимися периодами действия

### Получение данных о техкартах

```
GET /api/v1/dishes/{dishId}/technologicalCards
Headers: Authorization: Bearer {API_KEY}
```

Возвращает:
- Список всех техкарт для блюда
- Даты действия
- Состав (ингредиенты)
- Себестоимость

## ЯВКИ СОТРУДНИКОВ

### Получение информации о явках

**Endpoint:**
```
GET /api/v1/employees/{employeeId}/attendances
Parameters:
- dateFrom
- dateTo
```

**Возвращаемые данные:**
- ID смены
- Дата
- Время открытия смены
- Время закрытия смены
- Длительность
- Статус

### Запись явок в систему

#### Через BackOffice - Журнал явок

**Путь:**
Сотрудники → Явки → Журнал явок

**Работа с журналом:**
1. Выберите дату и сотрудника
2. Дважды щелкните на явку для редактирования
3. Отредактируйте:
   - Время открытия смены
   - Время закрытия смены
   - Тип явки (раб. часы, отпуск, больничный)

#### Через API

```
POST /api/v1/employees/{employeeId}/attendances
Body:
{
  "date": "2025-12-10",
  "openTime": "12:00",
  "closeTime": "20:00",
  "attendanceType": "WORKING_HOURS"
}
```

### Типы явок

1. **Работающий сотрудник** - обычная смена
2. **Отпуск** - оплачивается с коэффициентом, задаваемым в настройках
3. **Больничный** - оплачивается по коэффициенту
4. **Выходной** - не оплачивается или оплачивается по коэффициенту
5. **Дополнительная явка** - может быть добавлена вручную

### Настройка типов явок

**BackOffice → Сотрудники → Типы явок**

**Параметры:**
- Наименование
- Коэффициент оплаты (0-2.0)
- Описание

### Интеграция с системами учета рабочего времени

**Способ 1: Через API IIKO**
- Получать данные о явках из IIKO
- Передавать в 1С, зарплатную систему, Google Sheets

**Способ 2: Через СКУД (система контроля и управления доступом)**
- Терминалы по отпечатку пальца
- RFID карты
- Синхронизация с 1С через API

**Способ 3: Мобильный API**
- Для мобильного приложения HR
- Синхронизация явок в реальном времени

### Отчеты по явкам

**BackOffice → Сотрудники → Явки**

**Доступные отчеты:**
1. **Журнал явки** - история приходов/уходов
2. **Табель учета рабочего времени** - сводная таблица по месяцу
3. **Персональный отчет сотрудника** - отработанные часы и зарплата
4. **Сводный отчет по зарплате** - все начисления и удержания

## ПРИМЕРЫ ИНТЕГРАЦИЙ

### 1. Интеграция с Яндекс.Едой

**Шаг 1: Создание API-ключа** (см. выше)

**Шаг 2: Настройка в BackOffice**
1. Администрирование → Интеграции
2. Выбрать Yandex.Eats
3. Указать:
   - API-ключ IIKO
   - Префикс заказа (например, "YE_")
   - Рестораны, подключаемые к интеграции
   - Тип заказа ("Доставка")
   - Способы оплаты
   - Услугу доставки со свободной ценой

**Шаг 3: Выгрузка меню**
- Выбрать внешнее меню для Яндекс.Еды
- Убедиться, что все блюда имеют цены для этого сервиса

### 2. Интеграция с 1С (обмен данными)

**Способ 1: Файловый обмен**
1. BackOffice → Обмен данными → Выгрузка в 1С
2. Выбрать типы документов
3. Указать папку для выгрузки
4. В 1С загрузить файл через обработчик

**Способ 2: Прямая интеграция через API**
```
// Получить документы
GET /api/v1/documents/export?type=invoice&from=DATE&to=DATE

// Отправить документы в 1С
POST /api/v1/documents/import?system=1c
```

### 3. Интеграция с N8N (для Receptor)

**Основные операции в N8N:**

```
1. Получить данные о явках:
   GET /api/v1/employees/{id}/attendances

2. Получить отчет по продажам:
   GET /api/v1/reports/sales

3. Получить остатки:
   GET /api/v1/stock/report

4. Выгрузить в Google Sheets / Airtable
```

### 4. Интеграция с системой расчета зарплаты

**Данные, которые можно получить:**
- Явки сотрудников (дата, время, длительность)
- Типы явок (работа, отпуск, больничный)
- Ставки оплаты
- Начисления и удержания
- История зарплаты

### 5. Кастомная интеграция (пример на Python)

```python
import requests
import json
from datetime import datetime

# Конфигурация
API_KEY = "your_api_key_here"
ORG_ID = "organization_id"
BASE_URL = "https://api.iiko.ru/api/v1"

# Функция получения явок
def get_attendances(employee_id, date_from, date_to):
    headers = {'Authorization': f'Bearer {API_KEY}'}
    params = {
        'dateFrom': date_from,
        'dateTo': date_to
    }
    
    response = requests.get(
        f"{BASE_URL}/employees/{employee_id}/attendances",
        headers=headers,
        params=params
    )
    
    return response.json()

# Функция получения отчета по продажам
def get_sales_report(date_from, date_to):
    headers = {'Authorization': f'Bearer {API_KEY}'}
    params = {
        'organizationId': ORG_ID,
        'dateFrom': date_from,
        'dateTo': date_to,
        'groupBy': 'DAY'
    }
    
    response = requests.get(
        f"{BASE_URL}/reports/sales",
        headers=headers,
        params=params
    )
    
    return response.json()

# Пример использования
attendances = get_attendances("emp_001", "2025-12-01", "2025-12-10")
print(json.dumps(attendances, indent=2))
```
